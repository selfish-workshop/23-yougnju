# 예외 처리 정책 (Edge Cases)

**작성**: 기획팀
**참조**: PM 스펙 8장 + functional-spec.md
**범위**: 예상 가능한 예외 상황을 카테고리별로 분류하고, 각각의 처리 정책을 정의

---

## 0. 정책 원칙

이 서비스는 특이하게 **"데이터가 사라지는 것이 정상"**이다. 그래서 일반적인 웹앱에서 "에러"로 취급되는 많은 상황이 이 서비스에선 "정상 동작"이다. 그걸 흐리지 않는 것이 핵심 원칙.

1. **침묵의 원칙**: 사용자에게 경고·알림·에러 메시지를 최대한 적게 보여준다. 오직 본질적인 안내만
2. **비가역성 원칙**: 한 번 사라진 건 어떤 수단으로도 복구되지 않는다(회색 영역 없음)
3. **무충돌 원칙**: 사용자가 "뭘 잘못했나"라고 느끼지 않도록, 이상한 상태가 되면 조용히 초기화

## 1. 브라우저·디바이스 환경

### 1-1. localStorage 접근 불가
| 상황 | 프라이빗 모드, 쿠키·저장소 차단 설정, 구형 브라우저 |
|------|---|
| 감지 | `try { localStorage.setItem('test', '1'); localStorage.removeItem('test'); } catch {}` |
| 처리 | 메모리 내 상태(JavaScript 변수)로만 동작. 페이지 새로고침 시 기본값으로 초기화 |
| 사용자 공지 | **없음**. 슬라이더가 매번 5분으로 초기화되는 것 외에 사용자 불편 없음 |

### 1-2. JavaScript 비활성화
| 상황 | 사용자가 JS를 끔, 또는 구형·제한 환경 |
|------|---|
| 감지 | `<noscript>` 태그 |
| 처리 | "이 서비스는 자바스크립트가 필요합니다" 단순 안내 페이지 |
| 사용자 공지 | 짧은 한국어 안내 + 활성화 방법 힌트 |

### 1-3. 햅틱 API 미지원
| 상황 | iOS Safari(일부 버전), 데스크탑 브라우저 |
|------|---|
| 감지 | `'vibrate' in navigator` 체크 |
| 처리 | 햅틱 호출 시 조용히 무시. 다른 기능은 모두 정상 동작 |
| 사용자 공지 | **없음** |

### 1-4. `prefers-reduced-motion: reduce` 설정
| 상황 | 사용자가 움직임 감소 옵션을 켬 (접근성) |
|------|---|
| 감지 | CSS 미디어쿼리 또는 `window.matchMedia()` |
| 처리 | 잉크 번짐 대신 단순 페이드아웃 (500ms). 타이머 종료 로직은 동일 |
| 사용자 공지 | **없음** (이미 시스템 설정을 존중하는 것이기 때문) |

## 2. 네트워크·로딩

### 2-1. 네트워크 오프라인
| 상황 | 사용자가 비행기 모드, 지하철, WiFi 끊김 |
|------|---|
| 처리 | 정상 동작. **이 서비스는 네트워크가 필요 없음** (서버 통신 없음) |
| 예외 | 최초 페이지 로딩이 안 되면 당연히 접속 불가. 그 이후는 괜찮음 |

### 2-2. 페이지 로딩 도중 이탈
| 상황 | URL 클릭 후 페이지 다 뜨기 전에 뒤로가기 |
|------|---|
| 처리 | 브라우저 기본 동작. 아무 일도 일어나지 않음 |

### 2-3. 매우 느린 첫 로딩
| 상황 | 3G 등 느린 네트워크 |
|------|---|
| 처리 | 첫 화면이 보이기 전까진 빈 화면 + 로딩 인디케이터(필요 시 디자인팀 결정) |
| 최적화 | HTML/CSS/JS 최소화, 이미지 최소화(이상적으론 0), 폰트도 WOFF2 서브셋 |

## 3. 사용자 동작 예외

### 3-1. 아무것도 안 쓰고 대기
| 상황 | 캔버스 진입 후 10분째 빈 화면 |
|------|---|
| 처리 | **타이머 시작 안 함** (FN-03은 첫 입력 기준). 아무 일도 안 일어남 |
| 사용자 공지 | 없음. 정상 동작임 |

### 3-2. 타이핑 중 페이지 새로고침
| 상황 | 사용자가 실수로 F5 또는 당겨서 새로고침 |
|------|---|
| 처리 | 메모 내용 즉시 사라짐, 경고 없음 |
| 이유 | **의도된 동작**. 이 서비스는 "사라지는" 것이 본질이므로 브라우저 새로고침도 사라짐의 한 형태 |
| 디자인팀 협의 | `beforeunload` 경고를 띄울지 말지 → **띄우지 않는 것 권장**. 경고 팝업이 나오면 브랜드 철학과 충돌 |

### 3-3. 중간에 브라우저 탭 닫기
| 상황 | 사용자가 X 눌러서 탭 닫음 |
|------|---|
| 처리 | 메모 내용 즉시 사라짐, 경고 없음 |
| 이유 | 3-2와 동일 |

### 3-4. 뒤로가기 버튼
| 상황 | 브라우저 뒤로가기 |
|------|---|
| 처리 | 이전 페이지로 이동. 이 서비스 내 상태는 완전히 소실 |
| 이유 | 브라우저 기본 동작 존중 |

### 3-5. 탭 백그라운드 이동
| 상황 | 다른 탭으로 전환, 홈 화면으로 이동 |
|------|---|
| 처리 (제안) | 타이머는 `setTimeout`으로 계속 돔. 다만 모바일 OS가 백그라운드 탭을 절전 모드로 돌릴 수 있어 지연 가능 |
| 복귀 시 | `visibilitychange` 이벤트로 감지 → 현재 시각 기준으로 남은 시간 재계산 → 이미 종료됐어야 하면 즉시 사라짐 처리 |
| **🚨 PM 결정 필요** | `user-scenarios.md` 시나리오 5 참조 |

### 3-6. 매우 긴 글 입력
| 상황 | 사용자가 10만 자 이상 붙여넣기 |
|------|---|
| 처리 | 캔버스 자체에 길이 제한 없음. 브라우저 메모리가 허락하는 한 수용 |
| 예외 | 10만 자 초과 시 브라우저가 느려질 수 있음. 권장 한계 없음(소프트 제한) |
| 성능 이슈 | 매우 긴 텍스트의 잉크 번짐 애니메이션은 CPU 부담 → 디자인팀과 최적화 협의 필요 |

### 3-7. 매우 빠른 타이핑·입력
| 상황 | 사용자가 빠르게 연속 타이핑 |
|------|---|
| 처리 | 정상 동작. 햅틱은 첫 글자에만 발동 (매 키마다 진동하면 피로) |

### 3-8. 한국어 IME(입력기) 조합 중
| 상황 | "안녕" 입력 중 "안", "아ㄴ", "안ㄴ" 같은 조합 상태 |
|------|---|
| 처리 | 타이머 시작 기준을 `compositionend` 또는 `input` 이벤트로 맞춤 (구체는 구현팀 영역) |
| 주의 | 조합 시작을 "첫 입력"으로 볼지, 조합 완료를 "첫 입력"으로 볼지 결정 필요. **권장: 조합 시작** (빠르게 타이머 시작) |

## 4. 타이머 관련

### 4-1. 타이머 이미 진행 중에 추가 타이핑
| 처리 | 타이머 그대로, 영향 없음 (FN-03 정의대로) |

### 4-2. 슬라이더를 0 또는 음수로 설정 시도
| 처리 | UI 상 불가능 (최소값 30초 하드코딩). 만약 localStorage가 오염되어 이상한 값이 들어오면 기본값 300초로 덮어씀 |

### 4-3. 매우 긴 타이머(1시간)를 설정한 상태로 폰을 장시간 방치
| 처리 | 모바일 OS의 절전·탭 폐기 정책에 따라 타이머가 중단될 수 있음. 복귀 시 `visibilitychange` 재계산 로직이 처리 |

## 5. 데이터·저장소 관련

### 5-1. localStorage에 쓰레기 값이 있음 (이전 버전 잔재, 수동 조작 등)
| 감지 | 값을 읽을 때 타입·범위 검증 |
|------|---|
| 처리 | 불량 값은 즉시 기본값으로 덮어씀, 에러 없음 |

### 5-2. localStorage 용량 초과
| 상황 | 이론상 이 서비스가 저장하는 값은 매우 작음(3개 항목) → 거의 발생 안 함 |
|------|---|
| 처리 | 저장 실패 시 조용히 무시, 메모리 내 변수로만 동작 |

### 5-3. 사용자가 브라우저 데이터 삭제
| 처리 | localStorage도 함께 삭제 → 다음 방문 시 첫 방문자 취급 (FN-01 안내 다시 노출) |

## 6. 보안·프라이버시

### 6-1. 스크린샷 찍힘
| 상황 | 사용자가 자기 글을 스크린샷으로 남김 |
|------|---|
| 처리 | **막지 않음**. 웹에선 스크린샷 방지가 기술적으로 불가능하며, 막는다 해도 사진 찍으면 그만 |
| 브랜드 메시지 | "서버는 저장하지 않습니다"이지, "당신이 저장할 수 없습니다"가 아님 |
| 향후 고려 | 사용자 행동 연구 후 필요 시 "저장하지 마세요" 안내 추가 (CX팀 협의) |

### 6-2. 악의적 스크립트 삽입 시도
| 상황 | 사용자가 `<script>` 태그 등을 입력 |
|------|---|
| 처리 | 캔버스는 **플레인 텍스트만** 받음. HTML 렌더링 안 함 (contenteditable 사용 시 주의) |
| 구현 힌트 | `<textarea>`나 엄격한 sanitize 로직 사용 |

### 6-3. URL 파라미터로 초기 텍스트 주입 시도
| 상황 | `?text=악성텍스트` 같은 URL 공유 시도 |
|------|---|
| 처리 | URL 파라미터 **일체 무시**. 진입 시엔 항상 빈 캔버스 |
| 이유 | 1) 보안 2) 브랜드 일관성(누가 심어놓은 글로 시작하는 건 부자연스러움) |

## 7. 접근성 예외

### 7-1. 스크린 리더 사용자
| 상황 | 시각 장애인이 VoiceOver·TalkBack으로 접근 |
|------|---|
| 처리 | 캔버스·슬라이더·안내 모두에 적절한 ARIA 레이블 부여 |
| 주의 | 사라지는 애니메이션이 스크린 리더에게도 "사라졌음"을 알릴 수 있도록 live region 사용 |
| 결정 필요 | P0/P1/P2? 현재 P2로 분류(mvp-priority.md 참조), PM 재확인 필요 |

### 7-2. 색약·색맹
| 상황 | 다크 테마의 크림 글자 색상 대비 |
|------|---|
| 처리 | WCAG AA 이상 대비율 유지 (디자인팀 검증) |

## 8. 요약 — 사용자에게 보여야 하는 메시지 총 목록

이 서비스가 사용자에게 실제로 보여주는 UI 텍스트는 매우 적다. 목록:

1. 첫 진입 안내: "적어봐요. 5분 뒤 사라집니다." (1회)
2. 퇴장 안내: "오늘은 여기까지. 이제 자러 가볼까요?" (3초)
3. 슬라이더 값 표시: "⏱ 5분" 형식
4. (JS 비활성화 시) "이 서비스는 자바스크립트가 필요합니다"
5. 그 외 **없음**. 경고·에러·확인 팝업 일체 없음

**카피 총 4줄**. CX팀에서 최종 확정
